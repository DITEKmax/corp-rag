# ADR-001: Embedding model — bge-m3

- **Статус:** Accepted
- **Дата:** 2026-05-11
- **Затронутые компоненты:** Python AI Service (ingestion + retrieval), Qdrant

## Контекст

Системе нужна embedding-модель для построения векторного представления чанков документов. Корпус мультиязычный: основной язык — русский, второстепенный — английский (GitLab Handbook и подобные источники). Корпоративная лексика: процедуры, политики, должности, аббревиатуры.

Ограничения:
- Solo-проект, ~12 недель.
- Бесплатно (free tier API или локальный запуск на CPU).
- Желательно — без отдельного BM25-индекса; нужен полнотекстовый матчинг.
- Векторная БД — Qdrant, поддерживает hybrid (dense + sparse) в одной точке.

Кандидаты:
- **Qwen3-Embedding-8B** (рассматривался изначально в архитектурном отчёте)
- **BAAI/bge-m3**
- OpenAI `text-embedding-3-large` (платный, отбрасываем)
- Локальные модели типа `intfloat/multilingual-e5-large`

## Решение

**Используется `BAAI/bge-m3`** для эмбеддингов с двумя представлениями:
- **Dense** — 1024 измерения (нативная размерность модели)
- **Sparse (learned)** — `{token_id: weight}`, заменяет классический BM25

Оба вектора bge-m3 выдаёт **в одном прогоне** и оба хранятся в **одной точке** Qdrant.

Почему именно это:
1. **Multilingual из коробки** — bge-m3 заявлена на 100+ языков, на бенчмарках стабильно лучше Qwen3 на ru+en.
2. **Dense + sparse в одной модели** — экономим отдельный BM25-сервис (Elasticsearch не нужен), упрощаем пайплайн.
3. **MIT-лицензия** — без ограничений на использование.
4. **Контекст 8192 токенов** — достаточно для parent-chunks (~2000 токенов).
5. **Хорошая интеграция с Qdrant** — Qdrant нативно поддерживает hybrid query через RRF.

## Альтернативы

### Альтернатива A: Qwen3-Embedding-8B
- За: до 4096 измерений (через MRL), поддержка MRL — гибкая размерность.
- Против: только dense (нет встроенного sparse — пришлось бы вводить BM25 отдельно), хуже на мультиязычных бенчмарках.
- **Вердикт:** отвергнуто — потеря преимущества "dense+sparse в одной модели" перевешивает.

### Альтернатива B: OpenAI `text-embedding-3-large`
- За: высокое качество, простой API.
- Против: платно (на корпусе 30–50 документов всё ещё дёшево, но это операционная зависимость от вендора, и в дипломе важно показать on-premise возможность).
- **Вердикт:** отвергнуто — несовместимо с целью "может работать в закрытом контуре".

### Альтернатива C: `intfloat/multilingual-e5-large`
- За: хорошее качество на ru, локальный запуск.
- Против: только dense, размерность 1024 (как у bge-m3 — нейтрально), но **нет sparse-представления**.
- **Вердикт:** отвергнуто — bge-m3 покрывает тот же кейс плюс sparse.

## Последствия

### Что получаем
- Один embedding-инференс на чанк (dense+sparse одновременно) → экономия compute.
- Один индекс в Qdrant с двумя векторами → меньше сложности пайплайна.
- Multilingual качество для корпуса ru+en.
- Гибкость: можем экспериментировать с весами RRF без переиндексации.

### Чем платим
- bge-m3 тяжелее запускать локально на CPU (~2 GB модель). Решение: HuggingFace Inference API (бесплатный tier) или DeepInfra. См. ADR-009 (когда появится) по выбору хостинга.
- Размерность 1024 фиксирована для dense — нельзя экспериментально снижать (как было бы с Qwen3 MRL).
- bge-m3 sparse — менее изученное представление, чем классический BM25. Это решается тем, что классический BM25 остаётся как **baseline retriever** для ablation (см. EPIC 20.6 в ROADMAP).

### Что должно произойти если выбор окажется неправильным
Сигналы:
- На golden dataset bge-m3 sparse даёт заметно (≥5%) хуже recall@5, чем классический BM25.
- Локальный запуск слишком тяжёл, а HF Inference API стабильно даёт rate limit.

Стоимость переключения:
- Низкая: API одинаковый (bytes → vector). Можно переиндексировать корпус за час.
- Метаданные чанков в Qdrant не пострадают, только пересчитать векторы.
