# Phase 07.1: Fix Russian router and graph retrieval quality for RAGAS baseline - Context

**Gathered:** 2026-06-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 07.1 is a narrow remediation phase inserted after the committed Phase 07 Plan 06 RAGAS baseline. It fixes the Russian-corpus query-quality defects exposed by that baseline without mixing those fixes into the already-recorded baseline.

This phase improves answerability for the frozen 16-document Russian aviation/logistics corpus by fixing false router `UNSUPPORTED` outcomes, making graph aggregation retrieval Cyrillic-aware, repairing the live Neo4j corpus state for the golden documents, and adding enough eval metadata to diagnose future router decisions.

This phase does not weaken access filters, output guard behavior, citation validation, or production response contracts. It does not redesign multi-hop graph search into a full semantic path retrieval system; that larger graph-routing improvement is deferred to Phase 8 unless a small prerequisite is strictly needed to avoid breaking Phase 07.1 fixes.

</domain>

<decisions>
## Implementation Decisions

### Scope Split
- **D-386:** Phase 07.1 fixes three production quality defects now: Russian router classification, Cyrillic aggregation query terms, and live Neo4j corpus synchronization for the frozen golden corpus.
- **D-387:** Phase 07.1 also adds eval/report observability for `route_source` and `route_reason`, because the Phase 07 Plan 06 diagnostics could not reconstruct whether `UNSUPPORTED` came from rules, low LLM confidence, or classifier fallback.
- **D-388:** Full multi-hop Cypher redesign is deferred to Phase 8. The known issue is that current multi-hop retrieval is not text-conditioned and returns generic or duplicate relation evidence. That is real retrieval weakness, but it is larger than this baseline repair phase.
- **D-389:** Phase 07.1 may make only minimal multi-hop-adjacent safety changes if they are required by the router or metadata fixes. It must not attempt a broad entity-linking, semantic path ranking, or graph-RAG redesign.

### Router Fix
- **D-390:** Add Russian-aware routing rules for the frozen corpus patterns before the LLM fallback, especially aggregation and multi-hop questions that currently fall through English-only regexes.
- **D-391:** Keep the existing route set exactly as `FACTUAL`, `AGGREGATION`, `MULTI_HOP`, `COMPARISON`, and `UNSUPPORTED`; do not introduce new public route values.
- **D-392:** Keep `UNSUPPORTED` as a pre-retrieval short-circuit for genuinely unsupported or low-confidence questions. The fix is to avoid false `UNSUPPORTED` on answerable Russian corporate questions, not to retrieve for everything.
- **D-393:** Router changes must be tested against the five false-unsupported baseline records: `ru-aggregation-003`, `ru-multihop-002`, `ru-multihop-003`, `ru-multihop-005`, and `ru-multihop-006`.
- **D-394:** The router should remain rules-first with DeepSeek/OpenRouter JSON fallback. Do not replace it with a trained classifier or broad prompt-only routing.

### Graph Aggregation Fix
- **D-395:** Make aggregation query-term extraction Unicode-aware so Russian/Cyrillic tokens are retained instead of being dropped by `[A-Za-z0-9]+`.
- **D-396:** Keep graph aggregation access-filtered through Neo4j `Document` evidence, matching the existing Phase 5 invariant. Do not retrieve globally and filter in Python memory.
- **D-397:** Preserve citation requirements for aggregation answers. Counts, lists, and summaries still require document-backed evidence and valid returned citations.
- **D-398:** Add regression tests for Russian aggregation terms so empty `queryTerms` is no longer the default outcome for Russian questions.

### Neo4j Corpus Repair
- **D-399:** Clean Neo4j point-in-time state by `documentId`, not by dropping/recreating the database or collection. The six non-golden Phase 5/UAT leftovers to remove are: `Acme Remote Work Policy`, `TechCorp Approved Vendor List`, `TechCorp Phase 5 Query Policy`, `TechCorp Q1 2026 Security Incident Report`, `Politica`, and `Title`.
- **D-400:** Use the existing document-backed graph cleanup shape (`MATCH (d:Document {id: $documentId}) DETACH DELETE d` / `Neo4jGraphIndex.cleanup_document`) for the six extra Neo4j documents. Do not touch the 16 golden document IDs during cleanup.
- **D-401:** Rebuild graph data for the missing golden document `aa939285-1629-450d-b1f4-0e63ec6ba1c6` (`Политика планирования экипажей`) using the normal ingestion graph-building components rather than hand-writing synthetic Cypher entities or relations.
- **D-402:** The missing golden document must keep its existing document UUID because `golden_ru.jsonl` and `golden_ru.meta.json` are bound to live indexed UUIDs. Re-uploading as a new document ID is not an acceptable fix.
- **D-403:** Prefer a targeted maintenance/reindex path that replays or reconstructs a `document.uploaded` event for the existing Java document metadata and MinIO object, so Python rebuilds vector/parent/graph state and Java can receive an updated `document.indexed` event with non-zero `neo4jEntityCount`.
- **D-404:** If planning chooses a direct Python maintenance command instead of AMQP replay, it must still use existing parser/chunker/entity-extractor/graph-indexer components and must leave Java/Postgres `neo4j_entity_count` consistent with Neo4j. Silent live-state drift is not acceptable.
- **D-405:** Verification for corpus repair must prove Neo4j contains exactly the 16 document IDs from `ai-service/eval/golden/golden_ru.meta.json`, with no missing golden IDs and no extra non-golden IDs, and that Java Postgres reports non-zero `neo4j_entity_count` for all 16 golden documents unless a document legitimately extracts zero entities and is explicitly documented.

### Eval Metadata
- **D-406:** Add `route_source` and `route_reason` to eval query-phase records and JSON reports. These fields are diagnostic metadata; they should not require a public production contract change unless the existing response metadata already exposes them.
- **D-407:** If production `/v1/query` already has route source/reason internally but not in the REST response, Phase 07.1 should expose them to eval through a narrow eval-safe path or report-enrichment mechanism. Do not add frontend UI for these fields.
- **D-408:** Keep the existing query-phase cache behavior from Plan 07-06: collected query answers are written to disk before RAGAS scoring, and score-only can operate over that fresh cache.

### Post-Fix Evaluation
- **D-409:** After fixes, rerun one clean full RAGAS pass over the same frozen 40-record Russian golden dataset for comparison against baseline commit `4d23614`.
- **D-410:** The comparison must report at minimum answered count, `reranker_degraded_count`, faithfulness, answer relevancy, context precision, context recall, outcome accuracy, citation document recall, and route mix.
- **D-411:** The post-fix run should use the production `/v1/query` path, local `BAAI/bge-m3` embeddings for RAGAS answer relevancy, DeepSeek judge from `infra/.env`, topK 10, and sequential `concurrency=1`, unless planning discovers a concrete reason to adjust.
- **D-412:** The post-fix report is a comparison artifact. It must not overwrite the committed 07-06 baseline in a way that hides the before/after story.

### Invariants
- **D-413:** Do not weaken `OutputGuard`, citation validation, prompt-injection checks, or weak-evidence refusal thresholds to improve eval numbers.
- **D-414:** Do not broaden, bypass, or move access filtering out of Qdrant/Neo4j storage predicates. Retrieval quality fixes must operate inside the user-visible access boundary.
- **D-415:** Do not change default production `RetrievalOptions.topK` or add production retrieval-mode switches for this work. Eval-specific topK can stay in the eval harness.
- **D-416:** Do not alter the frozen corpus content, golden questions, reference answers, or expected document UUIDs as part of this remediation. The point is to compare system behavior on the same baseline dataset.
- **D-417:** Do not commit generated reports from the post-fix RAGAS run until the user validates that the run is clean and comparable.

### the agent's Discretion
- Choose exact Russian regex/tokenization implementation for router and aggregation if it is Unicode-aware, tested, and does not overmatch obvious out-of-scope prompts.
- Choose whether the targeted graph rebuild is implemented as an eval/maintenance CLI, an AMQP replay helper, or a small script, provided it preserves document UUIDs and Java/Neo4j consistency.
- Choose exact JSON field placement for eval-only `route_source` and `route_reason` if no public contract change is required.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `QueryRouter` already has rules-first routing and DeepSeek strict JSON fallback; Phase 07.1 can add Russian rules without replacing the architecture.
- `QueryGraphNodes` metadata already records `route_source`, `route_confidence`, and `route_reason` internally, but the committed eval report did not preserve these fields.
- `GraphRetriever` already has separate route handling for `AGGREGATION`, `MULTI_HOP`, and `COMPARISON`, so Cyrillic aggregation term extraction can be fixed locally.
- `graph_access_params()` and `document_access_predicate()` centralize Neo4j access-filter predicates and must be reused.
- `Neo4jGraphIndex.cleanup_document()` and `replace_document_graph()` provide the correct targeted graph mutation shape for cleanup/rebuild.
- The ingestion orchestrator already builds graph indexes from parsed/chunked documents and publishes `neo4j_entity_count`, but treats entity extraction/graph upsert failures as best-effort after vector indexing.

### Established Patterns
- Python owns RAG behavior and applies Java-provided access filters; Java owns auth, document metadata, and browser-facing APIs.
- `UNSUPPORTED` intentionally short-circuits before retrieval. Phase 07.1 fixes false routing, not that architecture.
- Retrieval metadata and diagnostics are report/debug surfaces, not user-facing feature expansions.
- Golden records are bound at document UUID granularity. Chunk hints are advisory only.
- Live model/eval work is run explicitly and not turned into hard CI gates.

### Integration Points
- Router tests should be added or extended around Russian aggregation and multi-hop questions in `ai-service/tests/test_query_router.py`.
- Graph aggregation tests should be added or extended in `ai-service/tests/test_graph_retriever.py` or a focused helper test for Cyrillic `queryTerms`.
- Query/eval tests should prove `route_source` and `route_reason` survive into cached `ragas_ru.json` records.
- Corpus-state repair should include live verification commands or a focused script that compares Neo4j `Document` IDs to `golden_ru.meta.json`.
- Post-fix RAGAS comparison should use the existing `ragas_runner.py` full-run path and preserve the pre-scoring cache behavior.

</code_context>
